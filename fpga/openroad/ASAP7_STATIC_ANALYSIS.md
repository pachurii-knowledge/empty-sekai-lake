# ASAP7 Static RTL Analysis — Single-Core niigo-lake OoO + SoC

> **Scope:** the single-core out-of-order RISC-V core + SoC as it enters ASAP7 synthesis — build config
> `RV64 + OOO_4WIDE + L1_CACHES + L1D_CACHE + AXI_MEMSYS + FPGA_BUILD + SYNTHESIS` (COHERENT=0; CCD/L2 param/`ifdef`-gated off).
> **Target flow:** OpenROAD *integrated* synthesis (yosys-slang frontend + generic ABC technology map) onto the
> ASAP7 7.5-track 7nm predictive PDK (RVT cells, TT corner, 1× collateral). Generic ABC has **no DSP macros, no
> register retiming, and no SRAM-macro mapping step** — those three gaps drive most of what follows.

> **Method:** a multi-agent static-analysis workflow — 11 per-subsystem analyzers, each finding adversarially
> verified against the cited RTL by a second agent, then ranked/synthesized — cross-referenced against the
> real per-block ASAP7 Fmax measured in the OpenROAD trial run (pre-CTS, placed parasitics). 31 timing findings
> and 36 cell-mapping findings survived verification.

> _Generated 2026-07-02. This is a design reference for the `asap7-synth` branch; it recommends RTL/flow changes
> but the numbers above the whole-core scale wall are structural lower bounds, not a routed/closed Fmax._


---

## Executive summary

As synthesized today the single-core niigo-lake OoO+SoC is bound at **~41.7 MHz by the CVFPU FP64 FMA**, whose 53×53→106b mantissa multiply and the following 164-bit CPA sit in one register-bounded stage (`fpnew_fma.sv:338/381`); the CVFPU DISTRIBUTED split (NumPipeRegs=2) never partitions the `*` operator, and generic ABC has neither DSP macros nor register retiming to fix it. The `ooo_mul_unit` 64×64 multiply (`ooo_mul_unit.sv:94-95`) is the same disease one tier up at ~103.6 MHz. The single biggest *area/mappability* risk is that OpenROAD integrated-syn has **no SRAM-macro step, so every inferred memory becomes flops**: L1 data ~262 Kbit, ITTAGE ~306 Kbit, TAGE ~46 Kbit, plus the 4096-bit 8R4W phys_reg_file and the 32×~348b ROB — together the dominant cell area *and* the source of async 1024:1 / 64:1 read-mux delay cones. The whole-core Fmax is unmeasured (ABC died mid-rewrite at 8.85M ANDs — the scale wall), so the 42 MHz floor is a structural-certain lower bound, not a routed number. Headline recommendation: **blackbox the L1 data/tag and register-then-SRAM the ITTAGE/TAGE tables, and replace the two monolithic multipliers (`*`) with operator-internally-pipelined Booth/Wallace CSA trees** — this attacks the binding timing path and the #1 area sink simultaneously; everything else (branch_stack, LSQ store-format, phys_reg_file mux) is second-tier below the 42 MHz floor.

## Recommended branch roadmap

Ordered by payoff-per-risk. Items 1-3 attack area+mappability with essentially zero functional risk; 4-5 attack the binding timing path with contained risk; 6+ are cleanup/flow.

1. **SRAM-macro blackbox the L1 data + L1 tag arrays.** The RTL is already textbook-SRAM-shaped (single sync read returning 4 ways, single byte-masked write per way). Map each L1 data way to an ASAP7 1RW 64×512 macro (OpenRAM/foundry compiler), L1 tag 1W2R→two 1RW copies, blackbox before ABC. **Payoff:** removes ~262 Kbit + ~40 Kbit of flops (the #1 area sink) and collapses the 64:1×512 read mux into a bitline access. **Correctness risk:** near-zero — module interface unchanged, behavior identical; verify byte-write-mask semantics match. **Bit-identical:** yes, single-core path unaffected (memory contents, not control).

2. **Replace the two monolithic multipliers with operator-internally-pipelined Booth/Wallace CSA trees.** `ooo_mul_unit` 64×64 and CVFPU FMA 53×53 each become Booth-radix-4 PP-gen + Dadda/Wallace reduction with an **explicit register between PP-generation and PP-reduction/CPA**, cutting at the existing S0/S1/S2 (mul) and INP/MID (FMA) boundaries. For the FMA the minimal first step is bumping ADDMUL `NumPipeRegs` (`niigo_fp_unit.sv:44`) so a reg lands mid-tree, but the coarse DISTRIBUTED split makes a dedicated pipelined multiplier module the reliable fix. **Payoff:** directly lifts the **41.7 MHz binding floor** and the 103.6 MHz mul path — the single highest-Fmax-leverage change. **Correctness risk:** medium — new multiplier RTL must be bit-exact vs the `*` result; validate against RV64G ACT M/D (289/289) and FP (D-fadd/fmul/fmadd) plus riscv-dv random vs Sail/Spike. Latency change ripples to scoreboard/wakeup timing — re-verify OoO scheduling. **Bit-identical:** architectural results identical; pipeline *depth* changes, so this is a microarchitectural (not COHERENT-gated) change — single-core semantics preserved, cycle counts differ.

3. **Register-then-SRAM (or shrink) the ITTAGE + TAGE tables.** Move the hashed index compute into a fetch-1 pipe stage so the array read is **synchronous**, do tag-compare/provider-select the next cycle, then map each 1024-deep bank to an ASAP7 SRAM; the async 1024:1 mux disappears. If a memory compiler is unavailable, cut INDEX_BITS (1024→256/512). **Payoff:** removes ~306+46 Kbit of async-read flops (largest predictor area) and takes the lookup off the fetch-cycle comb cone. **Correctness risk:** low-functional (predictions are hints, mispredicts recovered at branch resolve) but **adds a fetch-redirect latency cycle** — re-tune BP timing and confirm xv6 boot + branch-heavy riscv-dv still pass; also fix the power-up-X hazard (item 12) since flops no longer "power up to 0." **Bit-identical:** no (prediction timing shifts) — but architecturally transparent; keep off the COHERENT/CCD path (already excluded).

4. **branch_stack: split resolve/allocate into two pipe stages** (register `meta_premerge`/`valid_mask_premerge`), and register the 192b checkpoint restore mux. **Payoff:** lifts the 74.3 MHz frontend binder above the mul tier; unblocks whatever Fmax items 1-3 expose. **Correctness risk:** medium — checkpoint allocate/resolve is timing-sensitive to mispredict recovery; validate with riscv-dv branch-mask stress (the flow that found the `64daa82`/`80b6f86` deadlocks) + full ACT. **Bit-identical:** no (adds a cycle to checkpoint alloc) — microarch change, single-core semantics preserved.

5. **phys_reg_file + FP regfile: move the writeback bypass into a registered read stage and read FP sources at issue.** Bank/replicate read ports; drop the async-read requirement by adding one operand-read pipe stage. **Payoff:** removes the 18×64:1 + 4-way-compare cone from the read→execute path and the 12-way FP async read from rename. **Correctness risk:** high-touch — an added operand stage changes bypass timing across every FU; extensive ACT + riscv-dv re-verification. **Bit-identical:** no (pipeline depth) — do only if items 1-4 leave the RF cone binding.

6. **Latch/vendor/flow hygiene (low risk, do early alongside #1):** (a) swap `gated_clk_cell` passthrough for an ASAP7 ICG cell (or accept free-running FP flops); (b) `ifndef SYNTHESIS`-guard the niigo_memsys LCG fuzzer, the 64-bit cyc_q latency model, the perf-counter arrays, and the `initial $display`; (c) add a 2-FF reset-deassertion synchronizer; (d) reduce RAS_DEPTH 128→32; (e) rely on OpenROAD `repair_design -max_fanout` (not the inert Vivado attr) for abort_mask, and/or pre-replicate abort_mask per-consumer in RTL. **Payoff:** removes dead flops + wide sim-only arith from the netlist, fixes recovery/removal and X-propagation for tapeout. **Correctness risk:** minimal (sim-only or physical); the RAS-depth and buffering changes are functionally transparent. **Bit-identical:** RAS depth changes prediction only; the rest is synth-invisible.

7. **Flow-side options if the scale wall persists:** the whole-core ABC map died at 8.85M ANDs, so consider (a) an **external yosys + ABC with `-fast`/partitioned `synth` and `abc -dff`/retiming (`abc9`)** instead of integrated-syn, giving real register retiming that would materially help the multipliers/adders; (b) hierarchical/bottom-up synthesis (synthesize L1/predictor/FMA as pre-mapped macros, then the core) to dodge the flat-ABC blow-up; (c) config reduction for a first closing point — drop to RV32 (halves tag widths, mul/FMA sizes), fewer PHYS_REGS, smaller predictor tables. **Payoff:** a whole-core Fmax number at all, plus retiming for free on the `*`/CPA paths. **Risk:** flow-only, no RTL semantics change; RV32 config is a different verification target (RV32G 247/247), not the RV64 boot path.

## Verification & caveats

- **Structural-certain (verified by source inspection, independent of any Fmax number):** the two monolithic `*` operators (`ooo_mul_unit.sv:94-95`, `fpnew_fma.sv:338`), the async-read predictor arrays, the 8R4W phys_reg_file cone, the flop-mapping of all inferred memories (no SRAM step), the inert `max_fanout` attribute, the passthrough ICG cell, and the unguarded sim-only logic. These hold regardless of measurement.
- **Measured (pre-CTS placed parasitics, integrated-syn generic ABC):** only the **trustworthy self-contained blocks** — FMA 41.7, mul 103.6, div 199.8, branch_stack 74.3, free_list 102.4, mmu_tlb 264.6 (translate upper bound), ptw 655.6, rename_map 2229.9, busy_table 3506.6. The 42 MHz FMA is the lowest *real* block and the cited floor.
- **Do NOT quote as real numbers:** the standalone artifacts — int_issue_queue 0.9, tage_sc_l 1.3, active_list 2.5, load_store_queue 12.2, ooo_alu_pipe/dispatch/commit/fetch_decode 0/FAIL — are false combinational-loops through ports the core breaks externally, or const-fold to nothing in isolation. Their *structure* (ranked above) is real; their solo Fmax is meaningless.
- **Timeout (>2400s standalone, too big for solo ABC in 40 min):** l1_dcache, l1_icache, ittage_predictor, niigo_memsys — no isolated Fmax exists for these; their cones are ranked on structural depth only.
- **Scale wall / no whole-core number:** the full `riscv_core_ooo` slang-elaborates clean (0 err), ABC mapped 9.46M→8.85M ANDs then **died mid-rewrite** — there is **no measured whole-core Fmax**. The 42 MHz floor is therefore a *lower-bound binder*, not a routed/closed number; real whole-core Fmax will be at or below it once cross-module cones (abort_mask broadcast, translate chains, trap flush) and routing are included. FB2b's own observation that routed OOC ran ~5× worse than Quick-place on the Vivado/FPGA path corroborates that broadcast/route effects (rank 4) dominate once physical.
- **RV64 geometry drivers:** the 52-bit L1 tag comparator width and the FP64 53×53 / 64×64 multiply sizes are set by the RV64+Sv39 build; an RV32 config (roadmap #7c) materially shrinks all three. All numbers above are for the RV64 + OOO_4WIDE + L1_CACHES + L1D_CACHE + AXI_MEMSYS + FPGA_BUILD + SYNTHESIS single-core config (COHERENT=0, CCD/L2 param/ifdef-gated off).
